# TradeSkills 产品说明

本文面向业务同学、投资人和非底层研发同学，重点解释 TradeSkills 解决什么问题、产品价值是什么、典型使用方式是什么，以及当前阶段已经做到哪里。

## 1. 用一句人话说明它是什么

TradeSkills 是一个把“自然语言交易思路”变成“可运行、可回测、可周期执行的 Agent 策略”的平台。

用户不需要先写一整套量化代码，而是先写一份结构化 Markdown Skill，比如：

- 每 15 分钟执行一次
- 先扫描 OKX 永续合约
- 找出短期涨得过热的山寨币
- 结合 EMA、RSI、ATR、持仓量变化做判断
- 最多使用 10% 资金
- 必须设置止损

系统会把这份 Skill 变成一个可运行的 Agent，在两个场景下工作：

- `backtest`：回放历史时间点，模拟它过去会怎么做
- `live_signal`：按固定节奏运行，持续输出结构化交易信号

## 2. 它解决的核心问题

在很多交易团队里，策略从“想法”到“上线”之间通常有很长的断层：

1. 研究员或交易员有想法，但不会快速代码化
2. 量化/工程同学能实现，但迭代速度受限
3. 很多策略逻辑其实是半结构化的，不容易一开始就写死成规则引擎
4. 即使有想法，回测、复盘、调试、持续运行仍然要花很多工程成本

TradeSkills 想解决的是这个断层：

- 把策略表达门槛从“先写代码”降低为“先写自然语言 Skill”
- 用 Agent + 工具调用把策略运行起来
- 让策略先能验证，再逐步走向更严格的工程化

从业务角度看，它更像一个：

- `策略表达层 + Agent 决策层 + 回测验证层 + 定时运行层`

## 3. 现在的产品形态

当前版本是一个单机 demo 平台，包含三部分：

- Web Dashboard：看状态、上传 Skill、发起回测、查看最近信号
- API：保存 Skill、组织回测和实时任务、管理组合与状态
- Agent Runner：让模型读取 Skill、调用工具、输出结构化决策

也就是说，当前产品已经不是“概念验证 PPT”，而是能走通完整闭环的 demo：

- 上传 Skill
- 抽取策略契约
- 跑历史回测
- 生成交易决策
- 记录 trace 和组合变化
- 周期性运行 live signal

## 4. 对用户的核心价值

## 4.1 策略表达更快

最大的变化不是“更会下单”，而是“更快把策略想法表达出来”。

以前一个想法通常要经历：

- 口头描述
- PRD 或文档
- 工程翻译
- 编码
- 回测

现在可以先写成 Skill，直接进入可运行状态。

## 4.2 回测和实时使用同一份策略文本

传统流程里，回测版和实时版常常会有两套逻辑。

TradeSkills 当前强调的是：

- 同一份 Skill
- 同一套工具语义
- 两种运行模式

这对产品价值很重要，因为它降低了“研究版”和“运行版”之间的偏差。

## 4.3 策略行为可解释、可复盘

系统不是只给一个结果，而是会保存：

- 每一步做了什么决策
- 为什么这么做
- 调用了哪些工具
- 调用时看到的数据上下文
- 决策后组合如何变化

这让它天然适合：

- 策略复盘
- Agent 行为审计
- prompt / skill 版本对比
- 面向客户的过程解释

## 4.4 状态外部化，便于长期运行

Agent 本身不是靠上下文“死记住”过去发生过什么，而是把状态放到系统里统一管理。

这意味着：

- 每一轮执行都是可恢复的
- 回测和实时都能持续承接上一次结果
- 更容易做监控、恢复和审计

## 5. 现在最适合的使用场景

从产品阶段来看，当前最适合以下几类场景：

## 5.1 研究团队的策略原型验证

适合团队快速回答：

- 这个交易想法值不值得继续做
- 用自然语言能不能先表达清楚
- Agent 在有限工具条件下会不会稳定做出合理决策

## 5.2 面向客户或内部团队的演示平台

现在这套系统很适合 demo，因为它能把“Agent 如何分析市场、如何调用工具、如何产生决策”直观展示出来。

这比只讲一个抽象的 AI Trading Story 更容易建立信任。

## 5.3 策略协作的过渡层

很多时候研究员并不需要一上来就把策略写成完整量化程序。

TradeSkills 可以先充当一个过渡层：

- 先写 Skill
- 先跑历史
- 先看可解释结果
- 再决定要不要沉淀成更硬的程序化策略

## 6. 一个最典型的产品案例

最适合演示的样例，是仓库里自带的“短期过热做空 Skill”。

用户写的不是代码，而是这样的业务语言：

- 每 15 分钟运行
- 扫描 OKX USDT 永续
- 寻找短期涨得过热、筹码拥挤的币
- 结合 EMA、RSI、ATR、持仓量和资金费率判断
- 如果信号强，就用不超过 10% 的资金做空
- 必须设置止损

系统收到后，会做三件事：

1. 把文本变成结构化运行契约
2. 在历史模式下验证它过去会怎么做
3. 在实时模式下按固定节奏继续运行

这对业务演示特别有效，因为它把“策略想法 -> 运行结果”串成了一条完整链路。

## 7. 站在投资视角，这个项目的意义是什么

如果从投资或业务布局角度看，TradeSkills 的价值不只是一个回测工具，而是一个潜在的平台起点。

它有几个值得关注的方向：

## 7.1 它在做“策略表达层”

很多系统做的是执行层或数据层，但策略表达层往往仍然高度依赖人工工程翻译。

如果自然语言 Skill 能成为稳定接口，那么平台就有可能成为：

- 研究输入入口
- 策略协作入口
- 策略版本管理入口
- Agent 能力调度入口

## 7.2 它天然适合做“Agent 行为可视化”

AI 在交易领域最难的不是“让它说得聪明”，而是“让用户知道它为什么这么做”。

TradeSkills 现在已经具备可视化链条的基础：

- Skill 原文
- 工具调用
- 决策理由
- 组合变化
- 历史结果

这对 B 端产品化尤其重要。

## 7.3 它适合从单场景向平台化演进

虽然当前聚焦在交易 Skill，但底层模式本质上是：

- 自然语言规则
- Agent 决策
- 工具调用
- 结构化输出

这套模式未来并不局限于交易，也可以扩展到：

- 风控巡检
- 研究摘要
- 投研流程自动化
- 多角色协作决策

## 8. 需要诚实说明的当前边界

为了避免误解，当前阶段有几个边界需要明确讲清楚：

- 这是单机 demo，不是高并发生产系统
- 目前主要是回测和信号输出，不是实盘自动交易系统
- funding rate 和 open interest 还是 demo 级能力，不是完整交易所数据接入
- 实时模式当前更接近“定时读取最新快照”，不是事件驱动的真实实时流
- Agent 当前是单 Runner 模式，不是多 Agent 协同编排

这些边界不是缺陷，而是阶段性取舍：

- 当前优先验证产品闭环
- 再逐步增强数据真实性、执行可靠性和系统扩展性

## 9. 你可以怎么对外介绍它

如果对外只讲一段 30 秒的版本，我建议这样说：

> TradeSkills 是一个把自然语言交易策略变成可运行 Agent 的平台。用户上传一份 Markdown Skill，系统会自动抽取执行节奏、风险约束和工具需求，然后让 Agent 在统一工具框架下运行同一份策略文本，既可以做历史回测，也可以按周期输出实时信号，并完整记录每一步的决策依据和结果变化。

## 10. 给业务/投资同学的总结

如果用一句话概括当前产品价值，我建议这样讲：

> TradeSkills 不是在替代所有量化工程，而是在缩短“策略想法”到“可验证运行”的距离，把原本高度依赖人工翻译的策略表达过程，变成一个可运行、可复盘、可持续执行的 Agent 化产品流程。
