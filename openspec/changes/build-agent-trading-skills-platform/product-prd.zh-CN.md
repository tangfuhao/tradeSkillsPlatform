# Agent 交易 Skills 平台 PRD（V0.2）

## 1. 产品目标

构建一个面向现货加密市场的 Agent 交易 Skills 平台。用户以“文本 skill”的形式提交交易策略，平台将其标准化后接入历史行情模拟盘，在给定时间范围内回放交易行为，并输出收益、回撤、交易记录与可信度说明。

首版目标不是做实盘交易系统，而是做“策略验证平台”：
- 让用户可以快速验证一个文本策略是否有历史效果
- 让平台可以控制风险、成本与滥用
- 让结果具备复现性和解释性

## 2. MVP 范围

### 2.1 在范围内
- 现货加密市场回测
- 用户上传文本 skill
- 自动校验后立即运行最近 3 个月预览回测
- 人工审核后开放更大历史窗口
- 核心绩效指标展示
- 交易流水、订单流水、结果导出
- 平台侧历史数据管理与质量检查

### 2.2 不在范围内
- 实盘交易
- 自动生成策略
- 参数寻优
- Tick 级撮合仿真
- 多策略组合资金分配 UI
- 社交、收费、市场交易市场

## 3. 用户角色

### 3.1 策略作者
- 提交文本 skill
- 发起回测
- 查看结果
- 申请更大历史窗口

### 3.2 平台审核员/运营
- 审核 skill 是否可开放更大历史窗口
- 管理历史数据集
- 处理数据质量异常
- 观察队列、失败、风险事件

## 4. 核心业务规则

### 4.1 市场范围
- 首发市场：现货加密
- 市场形态：7x24 小时
- 盈亏记账：以 quote currency 计价

### 4.2 Skill 形式
用户上传的不是代码包，而是文本。推荐采用“YAML 元数据 + Markdown 正文”的格式。

建议文本结构：
- 基础元数据：名称、版本、市场、交易标的、时间周期、预热 bars、风险限制
- 策略描述：策略核心思想
- 入场规则
- 出场规则
- 仓位控制
- 失效条件
- 可调参数

### 4.3 运行权限分层
- 自动校验通过后：可立即运行最近 3 个月预览回测
- 人工审核通过后：可运行更大历史测试窗口

按 2026-04-13（Asia/Shanghai）计算，最近 3 个月的默认实现建议先按“滚动 90 天”处理，即约为 2026-01-13 到 2026-04-13。

### 4.4 回测对象模型
虽然首版用户体验是“单策略、单用户、单次回测”，但架构上需要拆开：
- Strategy Version
- Virtual Account
- Dataset Version
- Backtest Run

这样后续才能支持：
- 同一策略在不同账户参数下重复测试
- 同一策略在不同数据窗口下比较
- 多策略组合扩展

## 5. 用户主流程

### 5.1 提交并快速验证
1. 用户提交文本 skill
2. 平台做格式校验与策略标准化
3. 若通过，状态进入 `preview_ready`
4. 用户立即发起最近 3 个月回测
5. 平台返回预览结果

### 5.2 申请更大历史窗口
1. 用户选择某个已通过自动校验的 skill
2. 用户申请更大时间范围的历史回测
3. 平台进入人工审核流程
4. 审核通过后，skill 状态升级为 `approved_full_window`
5. 用户可运行更长历史区间

### 5.3 查看结果
用户可以看到：
- 核心指标
- 基准收益对比
- 交易流水
- 订单与成交记录
- 数据质量警告
- 预览/正式历史窗口标识

## 6. 核心功能设计

### 6.1 Skill 上传与版本管理
- 每次提交形成一个不可变版本
- 保留原始文本、解析结果、标准化策略 IR、校验报告、审核记录
- 新版本不会覆盖旧版本已产生的回测结果

### 6.2 Skill 标准化
平台不直接在每根 K 线上解释原始文本，而是先把文本转换为“标准化策略表示（Strategy IR）”。

这样做的原因：
- 结果更可复现
- 执行成本更低
- 更容易解释为什么下单
- 更容易做审核和错误提示

### 6.3 数据管理
- 平台侧维护标准化的现货加密历史数据
- 每份数据都有版本号
- 每份数据都有质量报告
- 回测必须绑定到具体数据版本，保证结果可重放

### 6.4 回测引擎
首版采用事件驱动 + Bar 级回放：
- 按时间顺序回放历史行情
- 先加载 warmup 数据
- 再逐步评估策略规则
- 计算订单、成交、仓位、资金、费用、滑点

### 6.5 Benchmark 设计
首版 benchmark 不要求用户必填，系统自动给默认值。

建议规则：
- 单标的现货回测：默认 benchmark = 同窗口下该标的 buy-and-hold
- 后续若扩展多标的：默认 benchmark = 等权 buy-and-hold 组合

这样既减少用户输入负担，也能让结果天然带有对照组。

## 7. 首版结果页核心指标

我建议首页优先展示这 8 个指标：
- Net PnL
- Total Return
- Benchmark Return
- Excess Return
- Max Drawdown
- Trade Count
- Win Rate
- Fees Paid

原因：
- 能回答“赚没赚钱”
- 能回答“是否跑赢被动持有”
- 能回答“回撤痛不痛”
- 能回答“交易是不是过度频繁”
- 能回答“手续费是不是吞掉收益”

次级指标如 Sharpe、Turnover、Exposure、Average Holding Time 可以放到第二层。

## 9. 技术架构建议

### 9.1 模块划分
- API 服务：用户上传、回测创建、结果查询
- Skill 标准化服务：文本解析、结构校验、IR 生成
- 回测 Worker：异步执行历史回放
- 数据服务：历史行情与数据版本管理
- 运营后台：人工审核、数据质量、运行监控

### 9.2 技术栈建议
- API：FastAPI
- Worker：Celery 或 Dramatiq + Redis
- 元数据存储：PostgreSQL
- 大文件与历史行情：S3/MinIO + Parquet
- 前端：Next.js

## 10. 风险点

### 10.1 文本 skill 语义不够明确
风险：用户写得太自由，平台无法稳定解释。
应对：强制模板 + 不支持即报错，不做静默猜测。

### 10.2 最近 3 个月结果容易“看起来很好”
风险：短窗口可能只反映阶段性行情。
应对：明确标记这是 preview result，不代表长期有效。

### 10.3 历史数据质量影响结果可信度
风险：缺 K、重复 K、精度不一致会让收益失真。
应对：每个数据版本都附带质量报告和 warning。

### 10.4 首版单策略单账户限制较强
风险：复杂策略用户会嫌不够用。
应对：在架构上提前把 Strategy / Account / Run 解耦。
